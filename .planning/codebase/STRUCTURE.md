# Codebase Structure

**Analysis Date:** 2026-04-09

## Directory Layout

```
PDF Unlocker/
├── services/           # Business logic layer
│   └── pdfService.js   # PDF processing service (WASM orchestration)
├── tests/              # Test suite
│   ├── pdfService.test.js # Unit tests for service layer
│   └── perf_test.js       # Performance benchmarking
├── ui/                 # Presentation layer
│   ├── app.js          # UI controller (DOM, events, queue)
│   └── styles.css      # Core styles (Aurora UI + Dark Mode)
├── .github/            # GitHub Actions (CI/CD)
├── .planning/          # Codebase documentation and plans
├── index.html          # Main application entry and CSP
├── sw.js               # Service Worker for offline support
├── sw-register.js      # Service Worker registration logic
├── manifest.json       # PWA manifest
├── package.json        # Dependencies and test scripts
└── vitest.config.mjs   # Test environment configuration
```

## Directory Purposes

**services/:**
- Purpose: Contains modules for specialized file processing logic.
- Contains: Vanilla JavaScript files with logic that doesn't touch the DOM directly.
- Key files: `services/pdfService.js`.

**ui/:**
- Purpose: Frontend presentation and styling.
- Contains: CSS for layout and styling, JS for DOM manipulation and user feedback.
- Key files: `ui/app.js`, `ui/styles.css`.

**tests/:**
- Purpose: Quality assurance through unit and performance tests.
- Contains: Vitest test files.
- Key files: `tests/pdfService.test.js`.

## Key File Locations

**Entry Points:**
- `index.html`: The only HTML entry point, loading CSS and deferred JS.
- `sw.js`: Controls offline caching and network strategies.

**Configuration:**
- `package.json`: NPM package configuration and scripts.
- `vitest.config.mjs`: Configures the Vitest test environment (jsdom).

**Core Logic:**
- `services/pdfService.js`: Encapsulates all QPDF-WASM logic and PDF validation.

**Testing:**
- `tests/pdfService.test.js`: Comprehensive test suite for the service layer.

## Naming Conventions

**Files:**
- JavaScript: camelCase (e.g., `pdfService.js`).
- CSS: lowercase (e.g., `styles.css`).
- Tests: `[filename].test.js`.

**Directories:**
- Plural lowercase (e.g., `services/`, `tests/`, `ui/`).

## Where to Add New Code

**New Feature (UI):**
- Implementation: Add DOM logic and events to `ui/app.js`.
- Styling: Add CSS rules to `ui/styles.css`.

**New Business Logic:**
- Service logic: Create a new file in `services/`.
- Tests: Create a corresponding `[service].test.js` in `tests/`.

**Global Config:**
- Scripts: Update `index.html` (e.g., adding external libraries).
- Metadata: Update `manifest.json` for PWA settings.

## Special Directories

**.planning/:**
- Purpose: Documentation generated by AI agents to map and plan the codebase.
- Generated: Yes.
- Committed: Yes.

**node_modules/:**
- Purpose: External dependencies (Vitest, JSDom, etc.).
- Generated: Yes (npm install).
- Committed: No.
